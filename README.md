# 会議招集の前に

**ちょっと待って。その会議、必要ですか？**

案件（A3・1枚）が主役。会議はA3の判断ブロックを埋める節目にすぎません。社内の問題解決を「案件（トヨタ問題解決8ステップ）」として1枚で俯瞰し、会議・レポート・チャットを報連相の原則で仕分けながら、意思決定と実行の速さ・質を可視化するシステムです。

**入り口はトップの「💬 チャット」タブです。** チャットに答えていくだけで、この件がレポート／チャット／会議のどれで済むかを一緒に見極めます。5往復以上やりとりすると「📋 会議招集まとめ」ボタンが出て、いつでも打ち切って会議招集の文面をこの画面のまま作れます。対話の内容は裏側で **トヨタA3 8ステップ／7つのムダ／報連相 仕分け** の参照3タブへ自動反映されます。最初の問いは **「案件名を教えてください。」** です。

**オンライン前提・AIあり前提。** バックエンドの LLM を有効にした状態を標準構成とし、対話もAIアシストも既定でONです。エンドポイントは配信元と同一オリジンを自動検出します。AIアシストは押したときだけ本文を送り、初回のみ「サーバーへ送信します（保存されません）」を1度だけお知らせして以後はワンクリックで動きます。サーバーは **ユーザー本文を保存・記録しません（ゼロリテンション）**。

> **技術注記:** `index.html` をブラウザで開くだけで、A3キャンバス・会議設計・レポート／チャットのテンプレートなど基本機能は動作します。各AIアシスト（R1〜R7）は応答が来なければ従来のルールベース処理へフォールバックします。ただし **入り口の💬チャット対話はAIアシストが必須**で、AI設定OFF時は開始できず、接続失敗時はルールベースへは切り替えずエラー＋「🔄 再試行」を表示します。

> **localStorage版は個人用デモです。** 作業データは各自のブラウザの中だけに保存され、端末間で同期しません。複数人で案件・会議・決定を共有するには、別途 **共有データストア**（サーバー側の永続化）が必要です。本リポジトリはその手前の、個人が試せるデモ実装です。

## 1. 名称と考え方

- システム名は **「会議招集の前に」**、タグラインは **「ちょっと待って。その会議、必要ですか？」**。会議を開く前に、それが本当に必要か（報告・連絡で済まないか）を問い直させることが出発点です。
- **案件が主役。** 継続的な問題解決を A3・1枚（トヨタ問題解決8ステップ）で表し、誰でも見に行ける一次情報源にします。トップ画面は「トヨタA3 8ステップ・キャンバス」です。
- **会議は独立イベントではない。** 会議（判断セッション）は A3 の判断ブロック（Step）を埋める **節目** にすぎません。案件を前に進める連続する PDCA の1コマとして扱います。
- **会議間に実行がある。** 会議で決めた Action を会議と会議の間に回し、次回は「前回アクションの持ち寄り」から始める連続性を仕組みで支えます。
- **決定は会議の専有物ではない。** 決定（Decision）は独立して管理し、メール・立ち話・上位判断など **会議を経由しない決定** も記録できます。

> **旧称の履歴:** 社内版は当初「会議関所（かいぎせきしょ）」、公開版は「会議ルーター」と呼んでいました。現在はいずれも「会議招集の前に」に統一しています（同一アプリ）。localStorage キー名の接頭辞 `sekisho.*` はデータ互換のため据え置いています。

## 2. 3原則

1. **会議は1分でも短い方がいい** — コストは「積み上がる」のではなく「予算から減っていく」。判断セッションは30分を標準とします。
2. **会議は参加者が少なければ少ないほどいい** — 「聞くだけ」の参加者はゼロにする。呼ばない代わりに申し送りが届く。
3. **報連相の統合** — 伝えたいことの時制で出口を分ける。

| 時制 | 種別 | 出口 |
|------|------|------|
| 過去のこと | 報告 | レポート（非同期） |
| 現在のこと | 連絡 | チャット（非同期） |
| 未来のこと | 相談 | 会議（同期） |

過去・現在の話は会議を開かず非同期で済ませ、**未来への問い（相談）だけが会議になれる**。案件トリアージでこの仕分けを最初に行い、報告・連絡に振り分けられた件は「本来開いていたはずの会議」の想定コストを **回避コスト** として記録します。

### トリアージの例外理由

「過去＝報告／現在＝連絡」は基本ルールですが、時制にかかわらず判断が必要になることがあります。トリアージ画面では以下の例外理由（複数選択）を選ぶと、判断セッション設計への導線が現れます（強制ではなく推奨。レポート／チャットで済ませることも可能）。

- 判断が必要 ／ 関係者間に認識差がある ／ 不確実性が高い ／ 緊急性が高い ／ 判断が不可逆 ／ 部門横断の調整が必要 ／ 決定権者の判断が必要

## 3. 8ステップ×報連相「推奨手段」マッピング

案件はトヨタ問題解決8ステップで構成され、各 Step には **推奨手段** があらかじめ決まっています。事実（過去）はレポート、実行状況（現在）はチャット、判断（未来）だけが会議です。

| Step | 名称 | 推奨手段 | 補足 |
|------|------|---------|------|
| 1 | 背景 | 🤝 会議 | 何に取り組むかを合意する |
| 2 | 現状把握 | 📄 レポート | 現状把握レポート（事実の層別・数字。原因や意見は書かない） |
| 3 | 目標設定 | 🤝 会議 | あるべき姿と目標値を決める |
| 4 | 要因解析 | 📄 レポート | なぜなぜ分析レポート（真因まで掘る） |
| 5 | 対策立案 | 🤝 会議 | どの対策を採るか決める |
| 6 | 対策実行・進捗共有 | 💬 チャット | 進捗連絡（対策の実行状況を非同期共有） |
| 7 | 効果確認 | 📄 レポート | 効果確認レポート（目標対実績を数字で） |
| 8 | 標準化 | 🤝 会議 | 決着・横展開を決め、A3・1枚レポートを生成 |

会議になるのは Step1・3・5・8（判断）だけで、事実共有の Step2・4・7 はレポート、進捗の Step6 はチャットが推奨です。

> **手段は「固定」ではなく「推奨」です。** どの Step にも、レポート・チャット・会議・そして **会議を経由しない決定（Decision）** を成果物として追加できます。報連相ナビは「次に埋めるべき Step とその推奨手段」を案内し続けますが、最終的な手段の選択は利用者に委ねます。

### 事実ゲート

判断の Step は、その直前の事実 Step が埋まっていることを前提にします。

- Step3（目標設定）← Step2（現状把握）
- Step5（対策立案）← Step4（要因解析）
- Step8（標準化）← Step7（効果確認）

事実 Step が未完了のまま判断セッションを設計しようとすると、「事実の説明に会議時間を使うことになります。先にレポートで共有しませんか？」という **警告** が出ます（開始はブロックせず、注意を促すのみ）。

## 4. 判断準備チェック（事実 Step の品質確認）

事実 Step（2 現状把握・4 要因解析・7 効果確認）には **判断準備チェック** があります。「入力済みか」ではなく「**次の判断に使える品質か**」をセルフチェックする8項目のチェックリストで、次の判断 Step へ進む前の品質ゲートとして働きます。

- 対象範囲が明確 ／ 数字または頻度がある ／ 対象期間がある ／ 正常と異常が比較されている ／ 情報源がある ／ 事実と意見が分離されている ／ 最終更新日が分かる ／ 判断に必要な事実が揃っている

チェックと合わせて **Evidence（判断根拠となる事実）** を記録できます。全項目をクリアすると、その Step の状態を「判断可能（ready）」へ遷移できます。全項目クリア前に ready にしようとすると、不足項目を挙げて促します。ダッシュボードの「判断準備不足件数」はこの未達を数えます。

## 5. 判断セッション設計（会議設計）

判断の Step は「判断セッション設計」で会議案内を組み立てます。必須・任意は次のとおりです。

**必須項目（埋めないと会議を開始できません）**

- **判断する問い** — このセッション全体で「何を決めるのか」を1つの問いにする。
- **決定者（決める人）** — 参加者に「決める人（決裁者）」の役割を最低1人割り当てる。
- **議題** — 「未来の問い（疑問形）」＋各議題の「今日決めること」。テーマ型（〜について／〜の件）や非疑問形は登録できません。
- **タイムボックス** — 議題時間の合計が30分を標準とし、超過分は「議題を分割して次回に回す」。

**任意項目（埋めると会議案内に反映され、会議が短く濃くなります）**

- 背景 ／ 選択肢（複数）／ 推奨案 ／ 判断基準 ／ 決めなかった場合の影響 ／ 今日の終了条件 ／ 案件（A3）とのStep紐づけ ／ 人時単価（値札に反映）

参加者の役割は「決める人／意見を出す人／情報を持っている人／聞くだけ」から選びます。「聞くだけ」は招集されず、会議終了後のレポート送付先へ自動的に回ります。値札（参加者×時間×単価）を見てから開く点は従来どおりです。

**このツールが対象としない会議**（守備範囲外）: 1on1・研修／講演・朝会・緊急インシデント対応・発散型ワークショップ・高度な機密案件。

## 6. 思考のコックピット（会議進行）

会議中はコックピットに30分フェーズバー（目的確認→事実共有→論点討議→決定→O/D/N確認）と予算ダウンカウンター、フェーズごとのコーチングヒントが表示されます。

- 議題ごとに決定を記録しないと次へ進めません。決定テキストの「何を」に曖昧語（検討する・調整する等）が入ると警告し、具体化を促します（もう一度押せばそのまま記録可）。
- 第2回以降はフェーズ0（前回アクションの持ち寄り確認）から始まります。
- 予定時間到達での延長は1回だけ。予算超過はアラートで示します。

---

## 7. 決定事項と申し送り（D/A/O/D/N）／会議外Decision

申し送りは議事録ではなく「次の状態」= **Decision / Action / Owner / Due / Next** の5点に絞って自動整形し、呼ばなかった人へコピー送付します。次回議題を決めて「次回会議を予約」すれば連続PDCAへ、Step8まで埋まれば「この案件を成果確定」で **A3・1枚レポート**（8ブロック）を生成します。

**決定（Decision）は独立エンティティ** として管理し、会議に属さない決定も扱えます。

- **会議を経由しないDecision登録** — メール・立ち話・上位判断など、会議を開かずに下した決定を Step 詳細から登録できます（判断 Step では特に推奨）。
- Decision には **判断根拠（basis）／採用しなかった案（rejectedOptions）／通知対象（notifyTo）** を添えられます。
- **決定履歴** は会議内外の Decision を時系列で一覧し、**🤝 会議で決定 / ✍️ 会議外で決定** のバッジで区別します。
- Step は done 後に **再調査（reinvestigate）／再オープン（reopened）** へ差し戻せます。再オープン後に再び決定した割合は、ダッシュボードの「再判断率」として見えます。

## 8. 画面構成（チャット主役＋参照3タブ）

トップに表示されるのは **「💬 チャット」** です。チャットで対話しながら、その内容が裏側で **トヨタA3 8ステップ／7つのムダ／報連相 仕分け** の3つの整理へ自動反映されます。ナビのタブは4つで、うち3タブはチャット内容を映す **読み取り専用の参照ビュー** です。判断セッション設計・思考のコックピット・決定事項と申し送り・フローダッシュボードなどの従来ツールは、**「トヨタA3 8ステップ・キャンバス」タブ内の「詳細ツール」** から開きます。

| タブ | 役割 |
|------|------|
| **💬 チャット**（トップ） | 対話の入り口。案件名から会議要否・最小メンバーを見極め、会議になる場合はこの画面のまま会議招集の文面まで作れます |
| **トヨタA3 8ステップ・キャンバス** | 全案件を1枚で俯瞰（読み取り反映＋詳細ツール）。報連相ナビ・事実ゲート・判断準備チェック・Step詳細（成果物／状態／履歴）・決定履歴。案件トリアージ／判断セッション設計／コックピット／フローダッシュボードへの導線もここ |
| **7つのムダ チェックリスト** | 対話から検知した「7つのムダ」の兆候（✅問題なし／⚠️注意／—未確認）を読み取り専用で表示 |
| **報連相 仕分けリスト** | 対話内容を 報告（過去）／連絡（現在）／相談（未来）の3レーンへ自動仕分けして表示。会議前に事前送付すべき資料には「会議前に事前送付」バッジが付く |

### チャットで進める（最初の入り口）

トップの **「💬 チャット」** が最初の入り口です。チャットに答えるだけで案件の下書きを作り、レポート／チャット／会議のどれが必要かを見極めます。

- **最初の問いは「案件名を教えてください。」** 案件名を入れると A3 の下書きが作られます。
- **対話はAIアシスト必須**です。AI設定がONのときだけ自由対話（`/api/v1/intake` によるステートレスなAI応答）を開始できます。AI設定OFF時は「AIアシストがOFFのため対話を開始できません。設定でONにしてください」と案内し、対話は始めません。サーバー不達・タイムアウト・エラー時は **ルールベースの質問ツリーへは切り替えず**、「⚠️ AIに接続できません。接続を確認して、もう一度お試しください。」＋「🔄 再試行」を表示します。再試行すると同じAIターンを再実行し、対話履歴・状態は保持されます。
- **5往復以上やりとりすると「📋 会議招集まとめ」ボタンが常設** され、いつでも対話を打ち切って、その時点の情報で会議招集の文面を **チャット内に出力** できます（コピー可・成果物として保存）。埋まっていない項目は省き、完璧を求めません。
- 対話で「事実が揃っていない」と分かった案件は、招集文面に **【会議までに事前送付のお願い】**（現状把握レポート＝Step2 の事前送付指示）が付きます。会議は説明ではなく判断に使う前提です。
- 会議招集の文面は **クライアント側の決定的ロジック** で組み立てます（LLMには任せません）。件名・背景・判断する問い・議題（10分）・招集メンバー（役割つき最小）・値札見込み・運用ルールを含みます。
- 会議になる場合は、判断セッション設計へ **判断する問い1件・タイムボックス10分** をプリフィルして渡すこともできます。

### 裏側の3フレームワーク（完璧を期さない）

対話の各応答は、裏側で **トヨタA3 8ステップ・7つのムダ・報連相** の3フレームワークへ best-effort で反映されます。ルールベースのヒューリスティックに加え、AIモードでは応答に添えられる構造化抽出（`extract`）を **分かった範囲だけ** 同じ器（参照3タブのデータ）へマージします。**完璧を期す設計ではなく**、対話が進むほど参照ビューが埋まっていく、という思想です。

### 基本フロー

```
💬 チャット（案件名 → 会議要否の見極め）
   ├─ 会議不要 → レポート／チャットで完了
   ├─ レポート先行 → 現状把握レポート（Step2）で事実を共有
   └─ 会議 → 5往復で「会議招集まとめ」→ 招集文面をチャット内に出力
             （必要なら判断セッション設計 → コックピット → 決定・申し送り）
```

対話を使わず、「トヨタA3 8ステップ・キャンバス」タブの詳細ツール（案件トリアージ等）や各案件カードから直接 Step へ入ることもできます。

## 9. フローダッシュボード

節約額ではなく、**案件の流れ（意思決定と実行の速さ）と停滞** を主役に見ます。データが無い指標は「—」で表示します。

### 案件の流れ（主役）

案件リードタイム（作成→成果確定）／判断待ち時間（Step ready→Decision）／DecisionからAction開始まで／当初期限内完了率／再判断率／目標達成率（Step7の目標対実績）／停滞案件数／判断準備不足件数。

### 会議削減効果（参考換算）

回避・短縮した会議を、解放した会議人時・業務時間と参考換算額で表します。

> **※ 参考換算額は実際の現金支出の削減額ではありません（社内レート×時間の目安です）。** 「削減した金額」ではなく「会議に費やさずに済んだ時間の目安」として読んでください。

### 速さの指標

アクション完了率／期限内完了率／議題の持ち越し率／会議の延長率／30分以内終了率／成果確定したA3案件数／期限超過アクション件数／持ち越し原因トップ。

### ムダ検知

各配列から毎回導出して計算します（累積値は持ちません）。詳しくは次章。

## 10. 「事技系職場の7つのムダ」対応マップ

「事技系職場の7つのムダ」ポスターの各項目を、システム機能で検知・抑止します（ポスター全文はダッシュボードの折りたたみに常設）。

| ムダ | システムでの対応 |
|------|-----------------|
| ① 会議のムダ（決まらない会議／決めない人も出る会議） | 決定を記録しないと議題を閉じられない・決裁者不在を警告。**決定ゼロ会議数** |
| ② 根回しのムダ（全員への事前回り） | 呼ばない人は招集せず申し送りを自動送付。「情報は自ら取りに行く」を明記 |
| ③ 資料のムダ（A4／A3一枚超の資料） | レポートは A3・1枚（約1500字）メーターで超過を警告。**A3・1枚超過レポート数** |
| ④ 調整のムダ（頑張っても進まない案件） | 停滞（持ち越し／期限超過）を検知しエスカレーションへ誘導。**停滞アクション（エスカレ候補）** |
| ⑤ 上司のプライドのムダ（「私は聞いていない」） | 申し送りの自動送付と「情報は自ら取りに行く」ルールで報告漏れを理由にした差し戻しを断つ |
| ⑥ マンネリのムダ（惰性で続ける業務） | 同一案件で決定ゼロ会議が続くと **マンネリ疑い** として警告。**マンネリ疑い案件数** |
| ⑦「ごっこ」のムダ（シャンシャン会議） | **v4で「異論ゼロ＝シャンシャン」の一律判定は廃止。** 異論の有無ではなく **判断の質** を検知：**代替案未検討の決定数／リスク未確認の決定数／判断根拠不足の決定数**。あわせて **期限変更回数の多いアクション**（理由つき）を、見積り前提の崩れや停滞の兆候として提示 |

> **v4でのムダ検知の見直し:** 「異論ゼロ会議率（シャンシャン）」という表面的な指標は、異論を演出するだけの逆効果を生みかねないため廃止しました。代わりに、決定そのものが「代替案・リスク・根拠を伴っているか」という **判断の質** を見ます。7つのムダポスター（全文）への言及は維持し、⑦の対応だけを新しい検知に置き換えています。

## 11. 技術構成

- 単一の `index.html`（HTML / CSS / JavaScript、依存ライブラリなし）。
- データ保存は `localStorage`。**個人用デモ**であり、複数人共有には共有データストアが必要です。
- スキーマは **v3**。起動時にマイグレーションが走ります。

### スキーマ・マイグレーション

起動時に `migrateSchema()` が冪等に走ります。

- **v1（`sekisho.schema` 未設定）**: `series` / `meetings` / `actions` を空配列で初期化。`sekisho.stats` / `sekisho.rate` は変換せずそのまま維持。
- **v2 → v3**: `series.steps.status` を7値へ（`todo`→`draft`→`review`→`ready`→`done`、done後に `reinvestigate` / `reopened`）、`filledBy`→`artifacts[]` へ、`history` を初期化。`actions` に当初期限（`dueOriginal`）・期限変更履歴（`dueChanges`）・着手時刻（`startedAt`）を補完。`meetings.agendas[].decision` を **独立 Decision** へ展開。`decisions` / `evidence` / `stakeholders` / `events` を用意。
- `stats` / `rate` / `llm` 設定は一切変換せず維持します。

### localStorage キー一覧（schema v3）

| キー | 内容 |
|------|------|
| `sekisho.schema` | スキーマバージョン（`"3"`） |
| `sekisho.rate` | 人時単価（円／時。既定 5000） |
| `sekisho.stats` | 累計統計（参考換算額・回避/開催件数・短縮・解放時間・呼ばずに済んだ人数・決定件数） |
| `sekisho.series` | 案件（A3）。タイトル・ゴール・8ステップの状態／成果物（artifacts）／履歴（history） |
| `sekisho.meetings` | 会議。議題・延長・所要・持ち越し原因など（直近50件を保持） |
| `sekisho.actions` | アクション。Owner・当初期限・期限変更履歴・着手時刻・状態・持ち越し回数 |
| `sekisho.decisions` | 決定（Decision）。会議内外を問わず独立管理。根拠・不採用案・通知対象・decidedAt |
| `sekisho.evidence` | 判断根拠となる事実（Evidence） |
| `sekisho.stakeholders` | 関係者 |
| `sekisho.events` | Event Log（追記型・直近500件を保持） |

速さの指標とムダ検知は各配列から **毎回導出** して計算します（累積値を持たない）。分母が0の率は「—」で表示されます。

### データ管理（エクスポート／インポート／バックアップ）

この端末のデータ（案件・会議・Decision・アクション・統計・設定）を1つの JSON に書き出し、別端末やバックアップとして持ち運べます。インポートは **現在のデータを全て上書き** し、直前状態への復元も可能です。schema 違いの JSON を取り込んでもマイグレーションを通して正規化します。

## 12. AIアシスト（公開Webサービス）

ブラウザ内で完結するアプリに、**AIアシスト用のステートレスAPI** を足し、公開Webサービスとしても配布できます。オンライン前提・AIあり前提の標準構成では **既定でON**、エンドポイントは配信元と同一オリジンを自動検出します。AIアシストは補助であって主役ではありません。データの主役は引き続き **ブラウザの localStorage** にあり、AIは「明示的に押したときだけ」対象テキストを送って手伝います。サーバーは **ユーザー本文を保存・記録しません（ゼロリテンション）**。

**送信の同意（v5-3で使い勝手を改善）:** 毎クリックの送信前プレビュー同意モーダルは廃止し、**初回のみ**「サーバーへ送信します（保存されません）」を1度だけ表示して以後は **ワンクリック送信** になりました（初回表示は `localStorage` の `sekisho.llm.firstNoticeShown` フラグで抑止）。従来どおり毎回内容を確認したい場合は、AIアシスト設定の **「送信前確認を毎回表示する」（既定OFF）** をONにすると送信前プレビューが復活します。AIをOFFにする、またはサーバーへ届かない場合でも、A3キャンバス・会議設計・テンプレートなどの基本機能とR1〜R7アシストのルールベース動作はそのまま使えます。ただし **入り口の💬チャット対話はAIアシスト必須** で、AI設定OFF時は開始できず、接続失敗時はルールベースへは切り替えずエラー＋「🔄 再試行」を表示します。

### アーキテクチャ

```
┌─────────────────────────── ブラウザ ───────────────────────────┐
│  index.html（アプリ本体・依存なし）                            │
│    ├─ 作業データ = localStorage（案件/会議/Decision/…）        │
│    └─ llmAssist(kind, payload)  ← 明示操作のときだけ送信        │
└───────────────┬───────────────────────────────────────────────┘
                │ POST /api/v1/<route>（対象テキスト＋最小context）
                ▼
┌──────────── server/router-api.mjs（Node標準ライブラリのみ） ────┐
│  静的配信:  GET / → index.html   /  GET /privacy → privacy.html │
│  ヘルス:    GET /healthz                                        │
│  保護:      32KBボディ上限 / 30req・分・IP / DAILY_LIMIT        │
│  ログ:      メタデータのみ（本文・応答は出さない）             │
│  ゼロリテンション: DB・ファイル書き込みなし                    │
└───────────────┬───────────────────────────────────────────────┘
                │ MOCK_LLM=1 なら外部送信せず決定的モック
                ▼
        OpenAI 互換 API（OPENAI_BASE_URL 宛。全ルート共通で OPENAI_MODEL / 既定 gpt-4o）
```

### AIアシスト一覧（対話インテーク＋R1〜R7・UI統合済み）

**R1〜R7のアシスト**は「明示的に押したとき」だけ動き、応答が来なければ従来のルールベース動作にフォールバックします。**対話インテーク**はAIアシスト必須で、応答が来ないときはルールベースへは切り替えずエラー＋「🔄 再試行」を表示します。すべてのルートは共通のモデル（環境変数 `OPENAI_MODEL`／既定 `gpt-4o`）を使用します。

| # | 機能 | エンドポイント | 概要 |
|---|------|----------------|------|
| intake | 対話インテーク | `POST /api/v1/intake` | チャットの入り口。案件名から会議要否・最小メンバーを見極め、ステートレスに応答（会話は保存しない） |
| R1 | 報連相オートルーター | `POST /api/v1/route` | 自由文を報告／連絡／相談に自動仕分けし、整形本文を返す |
| R2 | 議題解像度チェッカー | `POST /api/v1/refine-agenda` | 曖昧な議題を「問い」に磨き、決めるゴールと一言を返す |
| R3 | 曖昧アクション具体化 | `POST /api/v1/concretize-action` | 「誰が・何を・いつまで」の候補を抽出する |
| R4 | 申し送り要約 | `POST /api/v1/summarize-handoff` | 引き継ぎ内容を3行以内に要約する |
| R5 | A3・1枚コンパイラ | `POST /api/v1/compile-a3` | 入力を1500字以内のA3ドラフトに圧縮する |
| R6 | なぜなぜ深掘り | `POST /api/v1/five-whys` | 事象に対する「なぜ」候補を提示する（Step4） |
| R7 | ムダ診断 | `POST /api/v1/diagnose` | 現在の速さ・ムダ指標から改善提案を1段落で返す（フローダッシュボード） |

### 設定方法（要点）

| 変数 | 既定 | 意味 |
|------|------|------|
| `OPENAI_BASE_URL` | **（なし・必須）** | OpenAI 互換 API のベース URL。**デフォルト値なしの必須環境変数**。宛先は運用者が明示的に指定すること。未設定かつ非モックだと AI アシストは `503 llm_unavailable`（SSRF 対策）。 |
| `OPENAI_API_KEY` | （空） | OpenAI 互換 API キー。設定時のみ `Authorization: Bearer` を付与（任意）。 |
| `OPENAI_MODEL` | `gpt-4o` | 使用モデル。 |
| `PORT` | `8787` | 待受ポート。 |
| `DAILY_LIMIT` | `1000` | 1日あたりの LLM 呼び出し上限。 |
| `MOCK_LLM` | （空） | `1` で外部送信せず決定的モック応答。 |

```sh
# モックで起動（キー不要・外部送信なし）
MOCK_LLM=1 PORT=8787 node server/router-api.mjs
# → http://localhost:8787/ でアプリ、/privacy でプライバシー方針
```

ローカル/Docker の詳細な手順、コスト管理、ゼロリテンションの運用注意は [`docs/DEPLOY.md`](docs/DEPLOY.md) を参照してください。プライバシー方針の利用者向け説明は `privacy.html`（アプリのフッターからもリンク）にあります。

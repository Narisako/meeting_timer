# 会議関所（かいぎせきしょ） — A3案件ナビゲーター

**案件（A3・1枚）が主役。会議はA3の判断ブロックを埋める節目にすぎない。**

社内の問題解決を「案件（A3・トヨタ8ステップ）」として1枚で俯瞰し、会議・レポート・チャットを報連相の原則で仕分けながら、意思決定と実行の速さを可視化するシステムです。

デモ: `index.html` をブラウザで開くだけで動作します（サーバー・インストール不要、データはブラウザ内の localStorage に保存）。

---

## 1. 改訂コンセプト

従来版は「単発会議の効率化ツール」でした。本版は主役を **会議から「案件（A3・1枚）」へ** 移します。

- **案件が主役。** 継続的な問題解決を A3・1枚（トヨタ問題解決8ステップ）で表し、誰でも見に行ける一次情報源にします。
- **会議は独立イベントではない。** 会議は A3 の判断ブロック（Step）を埋める **節目** にすぎません。1回ごとに完結する催しではなく、案件を前に進める連続する PDCA の1コマとして扱います。
- **会議間に実行がある。** 会議で決めた Action を会議と会議の間に回し、次回は「前回アクションの持ち寄り」から始める——この連続性を仕組みで支えます。

参考にした[会議コストカウンター](https://s3-ap-northeast-1.amazonaws.com/mtg.design4all.biz/meeting_cost_counter.html)は「積み上がるコスト」を可視化するツールでした。本システムはそこから **案件のリードタイムと決定・実行の速さ** へと評価軸を移しています。

## 2. 3原則

1. **会議は1分でも短い方がいい** — コストは「積み上がる」のではなく「予算から減っていく」。会議は30分ハードブロック。
2. **会議は参加者が少なければ少ないほどいい** — 「聞くだけ」の参加者はゼロにする。呼ばない代わりに申し送りが自動で届く。
3. **報連相の統合** — 伝えたいことの時制で出口を分ける。

| 時制 | 種別 | 出口 |
|------|------|------|
| 過去のこと | 報告 | レポート（非同期） |
| 現在のこと | 連絡 | チャット（非同期） |
| 未来のこと | 相談 | 会議（同期） |

過去・現在の話は会議を開かず非同期で済ませ、**未来への問い（相談）だけが会議になれる**。関所（トリアージ）でこの仕分けを最初に行い、報告・連絡に振り分けられた件は「本来開いていたはずの会議」の想定コストを **回避コスト** として記録します。

## 3. 8ステップ×報連相マッピング

案件はトヨタ問題解決8ステップで構成され、各 Step は「どの手段で埋めるか」があらかじめ決まっています。事実（過去）はレポート、実行状況（現在）はチャット、判断（未来）だけが会議です。

| Step | 名称 | 埋める手段 | 補足 |
|------|------|-----------|------|
| 1 | 背景 | 🤝 会議 | 何に取り組むかを合意する |
| 2 | 現状把握 | 📄 レポート | 現状把握レポート（事実の層別・数字。原因や意見は書かない） |
| 3 | 目標設定 | 🤝 会議 | あるべき姿と目標値を決める |
| 4 | 要因解析 | 📄 レポート | なぜなぜ分析レポート（真因まで掘る） |
| 5 | 対策立案 | 🤝 会議 | どの対策を採るか決める |
| 6 | 実行計画 | 💬 チャット | 進捗連絡（対策の実行状況を非同期共有） |
| 7 | 効果確認 | 📄 レポート | 効果確認レポート（目標対実績を数字で） |
| 8 | 標準化 | 🤝 会議 | 決着・横展開を決め、A3・1枚レポートを生成 |

会議になるのは Step1・3・5・8（判断）だけで、事実共有の Step2・4・7 はレポート、進捗の Step6 はチャットへ回ります。A3案件ビューの「報連相ナビ」が、次に埋めるべき Step とその手段を案内します。

### 事実ゲート

判断の Step は、その直前の事実 Step が埋まっていることを前提にします。

- Step3（目標設定）← Step2（現状把握）
- Step5（対策立案）← Step4（要因解析）
- Step8（標準化）← Step7（効果確認）

事実 Step が未完了のまま判断会議を設計しようとすると、「事実の説明に会議時間を使うことになります。先にレポートで共有しませんか？」という **警告** が出ます（開始はブロックせず、注意を促すのみ）。会議の場で事実を説明し直すムダを、事前のレポートに追い出す仕掛けです。

## 4. トヨタ式30分会議の本質との対応

いわゆる「トヨタ式30分会議」（[東洋経済オンライン](https://toyokeizai.net/)などで紹介される会議手法）の勘所を、機能に落とし込んでいます。

| 本質 | 本システムの実装 |
|------|-----------------|
| 30分は「行動を変える制約条件」 | 議題タイムボックス合計が **30分を超えると開始不可**。超過分は次回へ分割する |
| 議題の解像度（今日決めること） | 議題は疑問形でしか登録できず、各議題に「今日決めること」を必須で添える |
| 参加者の事前準備（当事者化） | 主催者が全資料を用意するのではなく、参加者ごとに「期待する準備（持ち寄るもの）」を割り当てる |
| 説明は会議の外へ | 事実共有はレポート／チャットへ（報連相ルーターと事実ゲート）。会議は差分と判断だけ |
| D/A/O/D/N | 申し送りは議事録ではなく「次の状態」= **Decision / Action / Owner / Due / Next** の5点に絞る |
| 延長1回制と原因診断 | 予定時間到達で延長は **1回だけ**。未決議題は終了時に「持ち越し原因」を複数選択で診断してから閉じる |
| 評価は意思決定と実行の速さ | 評価軸は「会議時間の削減%」ではなく、アクション完了率・期限内完了率・案件リードタイムなど（第8章ダッシュボード） |

会議中はコックピットに30分フェーズバー（目的確認→事実共有→論点討議→決定→O/D/N確認）とフェーズごとのコーチングヒントが表示されます。決定テキストやアクションの「何を」に曖昧語（検討する・調整する等）が入ると警告し、具体化を促します。

## 5. 「事技系職場の7つのムダ」対応マップ

「事技系職場の7つのムダ」ポスターの各項目を、システム機能で検知・抑止します（ポスター全文はダッシュボードの折りたたみに常設）。

| ムダ | システムでの対応 |
|------|-----------------|
| ① 会議のムダ（決まらない会議／決めない人も出る会議） | 決定を記録しないと議題を閉じられない・決裁者不在を警告。ダッシュボードに **決定ゼロ会議数** |
| ② 根回しのムダ（全員への事前回り） | 呼ばない人は招集せず申し送りを自動送付。「情報は自ら取りに行く。『聞いていない』は禁句」を明記 |
| ③ 資料のムダ（A4／A3一枚超の資料） | レポートは A3・1枚（約1500字）メーターで超過を警告。**A3・1枚超過レポート数** を計測 |
| ④ 調整のムダ（頑張って調整しても進まない案件） | 停滞（2回持ち越し／期限超過7日超）を検知し、**エスカレーション会議** へ誘導。**停滞アクション数** |
| ⑤ 上司のプライドのムダ（「私は聞いていない」） | 申し送りの自動送付と「情報は自ら取りに行く」ルール文で、報告漏れを理由にした差し戻しを断つ |
| ⑥ マンネリのムダ（惰性で続ける業務） | 同一案件で決定ゼロ会議が2回連続すると **マンネリ疑い** として警告。**マンネリ疑い案件数** |
| ⑦「ごっこ」のムダ（シナリオどおりのシャンシャン会議） | 全議題が異論・懸念ゼロだと申し送りに注記。**異論ゼロ会議率（シャンシャン）** を計測 |

## 6. 画面フロー

タブは「① 関所」「② 会議設計」「③ コックピット」「④ 申し送り」「A3案件」「節約ダッシュボード」の6つ。会議は関所→設計→コックピット→申し送りの順で進み、②③④は前の段階を終えるまで無効化されています。

```
① 関所（報連相仕分け）
   ├─ 過去＝報告 → レポート雛形をコピー（回避コスト記録）
   ├─ 現在＝連絡 → チャット雛形をコピー（回避コスト記録）
   └─ 未来＝相談 → 単発 or「A3案件として始める」
        ↓
② 会議設計・会議案内
   議題（疑問形＋今日決めること）／役割つき参加者（期待する準備）／
   送付先／A3のStep紐づけ／値札 → 会議案内を自動生成
        ↓
③ コックピット
   30分フェーズバー＋予算ダウンカウンター／フェーズ別コーチング／
   第2回以降はフェーズ0（前回アクションの持ち寄り確認）から開始／
   議題ごとに決定を記録（曖昧語チェック）／延長1回制
        ↓
④ 申し送り（D/A/O/D/N）
   Decision / Action / Owner / Due / Next の5点を自動整形／
   次回会議を予約（連続PDCA）or「この案件は決着」／
   Step8決着ならA3・1枚レポート（8ブロック）を生成
        ↕
A3案件ビュー           節約ダッシュボード
8ブロックの俯瞰／        累計節約額・回避/短縮・
報連相ナビ／事実ゲート／   速さの指標／ムダ検知／
会議履歴・未完了アクション  7つのムダ ポスター
```

## 7. 使い方（A3案件を決着させるまで）

1. **関所で「相談（未来）」を選ぶ** → 「A3案件として始める」で案件（タイトル・ゴール）を作成。
2. **報連相ナビに従う。** 事実の Step（2・4・7）はレポート雛形、進捗の Step（6）はチャット雛形を埋めて保存。判断の Step（1・3・5・8）は「会議を設計する」へ。
3. **会議を設計する。** 議題を疑問形＋今日決めることで登録し、参加者に役割と期待する準備を割り当て、タイムボックス合計を30分以内に収めて会議案内を送付。
4. **コックピットで進行。** 30分フェーズバーとコーチングを見ながら、議題ごとに決定を記録。決められないときは「誰が・いつまでに・何を確認して再判断するか」を決定として残す。
5. **申し送りを共有。** D/A/O/D/N の申し送りを、呼ばなかった人へコピー送付。次回議題があれば「次回会議を予約」して連続PDCAへ。会議と会議の間にアクションを実行。
6. **次回はフェーズ0から。** 第2回以降は前回アクションの持ち寄り確認（全行回答するまで議題タイマーが開始しない）から始まる。
7. **Step8で決着。** 8ステップがすべて埋まったら申し送りで「この案件は決着」。8ブロックを整形した **A3・1枚レポート** が生成され、横展開の共有に使えます。
8. **ダッシュボードで振り返る。** 累計節約額に加え、アクション完了率・期限内完了率・案件リードタイム・各種ムダの検知件数を確認。

## 8. 技術構成

- 単一の `index.html`（HTML / CSS / JavaScript、依存ライブラリなし）。
- データ保存は `localStorage`。社内ポータルやファイルサーバーに置くだけで全員が使えます。
- スキーマは v2。起動時にマイグレーションが走り、v1 のキー（`sekisho.stats` / `sekisho.rate`）は変換せずそのまま維持します。

### localStorage キー一覧（schema v2）

| キー | 内容 |
|------|------|
| `sekisho.schema` | スキーマバージョン（`"2"`） |
| `sekisho.rate` | 人時単価（円／時。既定 5000） |
| `sekisho.stats` | 累計統計（節約額・回避/開催件数・短縮節約・取り返した分数・呼ばずに済んだ人数・決定件数） |
| `sekisho.series` | 案件（A3）。タイトル・ゴール・状態・8ステップの状態・次回下書き |
| `sekisho.meetings` | 会議。議題・決定・異論・延長・所要・持ち越し原因など（直近50件を保持） |
| `sekisho.actions` | アクション。担当（Owner）・期限（Due）・状態・持ち越し回数など |

速さの指標とムダ検知は各配列から **毎回導出** して計算します（累積値を持たない）。分母が0の率は「—」で表示されます。

## 9. v3: 会議ルーター（公開Webサービス＋AIアシスト）

v3 では、これまでブラウザ内で完結していたアプリに、**AIアシスト用のステートレスAPI** を足し、社内利用だけでなく公開Webサービスとして配布できる形にしました。公開版の呼称は **「会議ルーター」**、社内版の呼称は従来どおり **「会議関所」** です（同一アプリ）。

AIアシストは補助であって主役ではありません。データの主役は引き続き **ブラウザの localStorage** にあり、AIは「明示的に押したときだけ」対象テキストを送って手伝う位置づけです。サーバーは **ユーザー本文を保存・記録しません（ゼロリテンション）**。

### アーキテクチャ

```
┌─────────────────────────── ブラウザ ───────────────────────────┐
│  index.html（アプリ本体・依存なし）                            │
│    ├─ 作業データ = localStorage（案件/会議/アクション…）        │
│    └─ llmAssist(kind, payload)  ← 明示操作のときだけ送信        │
└───────────────┬───────────────────────────────────────────────┘
                │ POST /api/v1/<route>（対象テキスト＋最小context）
                ▼
┌──────────── server/router-api.mjs（Node標準ライブラリのみ） ────┐
│  静的配信:  GET / → index.html   /  GET /privacy → privacy.html │
│  ヘルス:    GET /healthz                                        │
│  保護:      32KBボディ上限 / 30req・分・IP / DAILY_LIMIT        │
│  ログ:      メタデータのみ（本文・応答は出さない）             │
│  ゼロリテンション: DB・ファイル書き込みなし                    │
└───────────────┬───────────────────────────────────────────────┘
                │ MOCK_LLM=1 なら外部送信せず決定的モック
                ▼
        Anthropic Claude API（Haiku / Sonnet を用途で使い分け）
```

### AIアシスト一覧（実装済み R1〜R6）

いずれも「明示的に押したとき」だけ動き、応答が来なければ従来のルールベース動作にフォールバックします。

| # | 機能 | エンドポイント | モデル | 概要 |
|---|------|----------------|--------|------|
| R1 | 報連相オートルーター | `POST /api/v1/route` | Haiku | 自由文を報告／連絡／相談に自動仕分けし、整形本文を返す |
| R2 | 議題解像度チェッカー | `POST /api/v1/refine-agenda` | Haiku | 曖昧な議題を「問い」に磨き、決めるゴールと一言を返す |
| R3 | 曖昧アクション具体化 | `POST /api/v1/concretize-action` | Haiku | 「誰が・何を・いつまで」の候補を抽出する |
| R4 | 申し送り要約 | `POST /api/v1/summarize-handoff` | Sonnet | 引き継ぎ内容を3行以内に要約する |
| R5 | A3・1枚コンパイラ | `POST /api/v1/compile-a3` | Sonnet | 入力を1500字以内のA3ドラフトに圧縮する |
| R6 | なぜなぜ深掘り | `POST /api/v1/five-whys` | Sonnet | 事象に対する「なぜ」候補を提示する |

> R7（指標からのムダ診断）は、サーバーAPI（`POST /api/v1/diagnose`）としては存在しますが、UI統合は次のイテレーションで行います。上表には実装済み（UI統合済み）の R1〜R6 のみを記載しています。

### 設定方法（要点）

| 変数 | 既定 | 意味 |
|------|------|------|
| `ANTHROPIC_API_KEY` | （空） | Claude API キー。未設定かつ非モックだと AI アシストは 503。 |
| `PORT` | `8787` | 待受ポート。 |
| `DAILY_LIMIT` | `1000` | 1日あたりの LLM 呼び出し上限。 |
| `MOCK_LLM` | （空） | `1` で外部送信せず決定的モック応答。 |

```sh
# モックで起動（キー不要・外部送信なし）
MOCK_LLM=1 PORT=8787 node server/router-api.mjs
# → http://localhost:8787/ でアプリ、/privacy でプライバシー方針
```

ローカル/Docker の詳細な手順、コスト管理、ゼロリテンションの運用注意は [`docs/DEPLOY.md`](docs/DEPLOY.md) を参照してください。プライバシー方針の利用者向け説明は `privacy.html`（アプリのフッターからもリンク）にあります。
